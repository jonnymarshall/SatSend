# v1.4.16 — Invoice Number Character Limit: Manual Test Guide

## What this version does

Caps `invoice_number` at **30 characters** across the whole pipeline:

1. **DB CHECK constraint** (`0020_invoice_number_length.sql`) — `invoice_number is null or char_length(invoice_number) <= 30`. Migration also deletes any pre-existing rows over 30 chars (zero offenders on remote at apply time).
2. **Server actions** — `saveDraft` and `updateDraft` reject 31+ chars with `Error("invoice_number: Invoice number must be 30 characters or fewer.")` before the DB call. The form already routes `invoice_number:` errors back to the field via the existing `parseServerError` plumbing.
3. **Form input** — `<input maxLength={30}>` on `#input-invoice-number`, plus a live `N / 30` counter under the field (`#counter-invoice-number`).
4. **`duplicateInvoice`** — old `${source} (copy)` suffix replaced with `buildDuplicateInvoiceNumber()`. Always appends `... (copy)` (10 chars). Trims source from the end only when total would exceed 30. Examples: `INV-001` → `INV-001... (copy)` (17 chars, no trim). 30-char source → trims to first 20 chars + `... (copy)` = exactly 30.

The motivating bug was ugly subject lines like `Invoice ABCDEFGHIJKLMNOPQRSTUVWXYZ-2026-04-29-FOLLOWUP-V2 from …` — TEST 5 confirms that path.

> **Note on the server-side guard:** `assertInvoiceNumberLength` in `saveDraft` / `updateDraft` is covered by 4 unit tests in `src/app/(dashboard)/invoices/actions.test.ts` (search for `v1.4.16`). It can't be reliably exercised from the browser — the form's `maxLength` clips pastes before React's `onChange` fires, so any DevTools "edit attribute" bypass tends to snap back on re-render. The DB CHECK in TEST 3 below is the actual safety net for non-form code paths.

## Quick smoke test (~2 minutes)

If you only have time for 3 tests, run TEST 1, TEST 2, and TEST 5. They cover: form-level cap discoverability, end-to-end save with a 30-char value, and the original motivating UX bug.

---

## TEST 1 — Form blocks typing past 30 chars and live counter is visible

**Proves:** form-level cap + UX discoverability (counter).

**Setup:** open `/invoices/new`.

**Run:**
1. Click into the **Invoice number** field.
2. Note the counter under the field reads `0 / 30`.
3. Start typing — confirm counter updates each keystroke (`1 / 30`, `2 / 30`, …).
4. Try to keep typing past 30 chars (mash a key for a few seconds).

**Expect:**
- The field's value never exceeds 30 characters (`maxLength={30}` blocks further input).
- The counter caps at `30 / 30` and stays there.
- No console errors.

---

## TEST 2 — Save and publish an invoice with a 30-char invoice number

**Proves:** 30-char boundary works end-to-end through save + publish + display surfaces.

**Setup:** open `/invoices/new`. Have a fresh testnet BTC address ready.

**Run:**
1. Type `ABCDEFGHIJKLMNOPQRSTUVWXYZ-123` (exactly 30 chars) into the **Invoice number** field. Counter should read `30 / 30`.
2. Fill required fields (`client_name`, at least one line item, BTC address).
3. Click **Save draft** → expect navigation to `/invoices/<id>/edit`. No errors.
4. Reload the page → the field still reads the full 30-char string.
5. Visit `/invoices` (the dashboard list) → confirm the row's Invoice column shows the full 30-char string. Note any column-width pressure on adjacent columns; the test passes if nothing is *clipped*.
6. Click into the invoice → confirm the detail page renders the number in its heading without clipping or wrap weirdness.
7. From the detail page, click **Publish** (publish-only path; don't send email yet).
8. Open the public payer page (`/invoice/<id>`) in a new tab → confirm the heading reads `Invoice ABCDEFGHIJKLMNOPQRSTUVWXYZ-123`. Page renders cleanly.
9. From the detail page, click **Download PDF** (or whatever the dropdown is labelled). Open the PDF → confirm the invoice-number value renders cleanly without clipping in the PDF header.

**Expect:** all six surfaces (form, list cell, detail heading, public-page heading, PDF header, plus the post-reload form re-render) display the 30-char value without truncation, clipping, or layout breakage.

**DB confirmation (optional):**
```sql
select id, invoice_number, char_length(invoice_number) as len
from invoices
where char_length(invoice_number) = 30
order by created_at desc
limit 5;
```

## TEST 3 — DB CHECK constraint rejects direct SQL bypass

**Proves:** the safety net at the DB layer survives a totally bypassed application stack (e.g. someone running ad-hoc SQL).

**Setup:** access to remote SQL editor (Supabase Studio for the linked SatSend project, or `psql` against the connection string).

**Run:**
```sql
-- Pick any of your existing user_ids
insert into invoices (user_id, invoice_number, status, line_items, tax_percent, subtotal_fiat, tax_fiat, total_fiat, currency, your_email, client_name, client_email)
values (
  (select id from auth.users limit 1),
  'X1234567890123456789012345678901',  -- 31 chars
  'draft',
  '[]'::jsonb,
  0,
  0,
  0,
  0,
  'USD',
  'test@example.com',
  '',
  ''
);
```

**Expect:** the insert fails with a constraint-violation error mentioning `invoice_number_length`. Cleanup is not needed (the row was never written).

Then, sanity-check the constraint exists:
```sql
select conname, pg_get_constraintdef(oid)
from pg_constraint
where conrelid = 'invoices'::regclass
  and conname = 'invoice_number_length';
```

Should return one row with `CHECK ((invoice_number IS NULL) OR (char_length(invoice_number) <= 30))`.

## TEST 4 — Duplicating a long-but-legal invoice still saves cleanly

**Proves:** `duplicateInvoice` + `... (copy)` suffix is constraint-safe even at the boundary. Without the v1.4.16 change, this would silently throw a CHECK-constraint error mid-action.

**Setup:** create or pick an existing invoice with `invoice_number` of exactly **25 characters** (e.g. via TEST 2's flow but with a 25-char value). Note its name.

**Run:**
1. From `/invoices`, click the duplicate action (kebab → Duplicate, or whatever the row action is named) on the 25-char invoice.
2. You're redirected to `/invoices/<new-id>/edit`.
3. Look at the Invoice number field on the duplicate.

**Expect:**
- The new invoice's number is `<first-20-chars-of-source>... (copy)` — exactly 30 characters total. Source's last 5 chars are dropped to make room.
- The counter reads `30 / 30`.
- No errors in console.
- DB confirmation:
  ```sql
  select invoice_number, char_length(invoice_number) as len
  from invoices
  order by created_at desc
  limit 1;
  ```
  Length must be exactly 30.

Bonus: also duplicate a short invoice (e.g. `INV-001`, 7 chars). The new one should be `INV-001... (copy)` (17 chars, no trim) — confirms short names also get the new suffix.

## Rollback notes

If you need to roll back the constraint without rolling back code:
```sql
alter table invoices drop constraint invoice_number_length;
```
This is safe because the form, server actions, and `duplicateInvoice` keep producing values ≤ 30 chars even without the DB constraint. The constraint is the safety net, not the only line of defence.
