## v1.4.18 , Resend Webhook: Sent vs Delivered vs Bounced , Manual Test Guide

## What this version does

Closes the gap between "Resend accepted the send request" (`status='sent'`) and "the recipient mailbox confirmed receipt" (`status='delivered'`). Adds three new `email_events.status` values:

- `delivered` , Resend's webhook reports the recipient mailbox accepted the email.
- `bounced` , recipient mailbox rejected delivery (bad address, full mailbox, etc.). `error_message` captures the reason.
- `complained` , recipient marked the email as spam.

Three pieces wired up:

1. **`POST /api/webhooks/resend`** , verifies the Svix-signed payload from Resend, deduplicates by `svix-id` via a new `webhook_deliveries` table, and applies the lifecycle update on the matching `email_events` row.
2. **Activity card** (`/invoices/[id]`) , the email rows now read `awaiting delivery` while `status='sent'`, `delivered` (green mail-check icon), `bounced` (red mail-x icon, with reason), or `marked as spam` (orange mail-warning icon).
3. **`/invoices` per-row indicator** , the red `AlertCircle` that v1.4.9 added for `failed` rows now also fires for `bounced` and `complained` rows, with per-status tooltip text.

Out of scope: `email.opened` / `email.clicked` events (read-receipt territory), retry-on-bounce UX, dashboard counters.

---

## Setup

You will need:

- The dev server running on a **publicly reachable URL** (Resend cannot reach `localhost`). Use a Vercel preview deploy, or `ngrok http 3000` against a local dev server.
- `RESEND_WEBHOOK_SECRET` set in the env where the route runs. (See README , Email setup section.)
- A webhook endpoint configured in the Resend dashboard pointing at `https://<your-domain>/api/webhooks/resend`, subscribed to `email.sent`, `email.delivered`, `email.bounced`, `email.complained`.
- A logged-in dashboard tab.
- One draft invoice with a `client_email` you can publish-via-email.

## Quick smoke test (~3 minutes)

If you only have time for two tests, run **TEST 1** (golden path: sent , delivered) and **TEST 2** (bounce path). They cover the two ends of the lifecycle that this branch unlocks.

---

## TEST 1 , Golden path: `sent` flips to `delivered` within seconds

**Proves:** the webhook accepts a valid Resend payload, looks up the matching `email_events` row by `resend_message_id`, and advances the lifecycle to `delivered`. The Activity card reflects the change.

**Setup:** open the draft invoice, fill in your own real email address as `client_email` (so you can verify delivery yourself).

**Run:**
1. Click `Publish , send via email`. The Activity card should immediately show `Invoice published , awaiting delivery , <your-email>`.
2. Wait ~10 seconds, then refresh the page (or open it in a new tab).

**Expect:**
- The Activity row label is now `Invoice published , delivered , <your-email>`.
- The icon is the green mail-check icon (`data-icon="mail-check"`).
- A real email arrived in your inbox.

**DB confirmation:**

```sql
select status, resend_message_id, updated_at
  from email_events
 where invoice_id = '<your-invoice-id>'
   and email_type = 'invoice_published'
 order by created_at desc
 limit 1;
-- status should be 'delivered'
```

---

## TEST 2 , Bounce path: `sent` flips to `bounced` with reason captured

**Proves:** a hard-bouncing recipient address surfaces as `bounced` in both the Activity card and the `/invoices` per-row indicator, with the bounce reason in `error_message`.

**Setup:** create or duplicate a draft invoice. Set `client_email` to `bounce@simulator.amazonses.com` (a recognised hard-bounce test address).

**Run:**
1. Click `Publish , send via email`.
2. Wait ~15 seconds (bounces take a touch longer than deliveries), refresh.

**Expect:**
- Activity card label: `Invoice published , bounced , bounce@simulator.amazonses.com`. Mail-x icon (red).
- Below the row, a short red error line with the bounce reason (e.g. `mailbox does not exist`).
- On `/invoices`, the row now shows the red `AlertCircle` indicator next to the status badge. Hovering the indicator shows the tooltip `Email bounced`.
- The Mail icon for `send_method='email'` is replaced by the AlertCircle (not stacked).

**DB confirmation:**

```sql
select status, error_message
  from email_events
 where invoice_id = '<your-invoice-id>'
   and email_type = 'invoice_published'
 order by created_at desc
 limit 1;
-- status='bounced', error_message contains the bounce reason
```

## TEST 3 , Complaint path: spam-mark flips status to `complained` (DEFERRED)

**Status:** Deferred manual verification. There is no clean way to trigger `email.complained` without a real recipient marking a delivered email as spam in their inbox client. Both code paths are fully covered by unit tests:

- `route.test.ts` , `flips status to 'complained' for email.complained`
- `invoice-activity-card.test.tsx` , `renders a 'complained' email as 'marked as spam' with the mail-warning icon`
- `data-table.test.tsx` , `renders the failed-email indicator for a complained row with a 'marked as spam' tooltip`

The first time a real spam complaint comes through production, manually verify the UI shows the orange mail-warning icon, the `marked as spam` label, and the `/invoices` indicator tooltip , then tick this test off retroactively.

## TEST 4 , Idempotency: replaying the same event is a no-op

**Proves:** the `webhook_deliveries(svix_id pk)` dedupe table guarantees at-most-once processing even if Resend retries the same event.

**Setup:** capture a real Resend webhook request from your endpoint's recent activity in the Resend dashboard (`Webhooks , <endpoint> , recent attempts , click an attempt , copy raw request`). Save the body to `body.json`, and note the `svix-id`, `svix-timestamp`, and `svix-signature` header values.

**Run:**
1. Confirm the matching `email_events` row's `updated_at` timestamp:

   ```sql
   select id, status, updated_at from email_events where resend_message_id = '<id>';
   ```

2. Replay the exact request with curl:

   ```bash
   curl -X POST https://<your-domain>/api/webhooks/resend \
     -H "svix-id: <svix-id>" \
     -H "svix-timestamp: <svix-timestamp>" \
     -H "svix-signature: <svix-signature>" \
     -H "Content-Type: application/json" \
     --data-binary @body.json
   ```

3. Re-query the row.

**Expect:**
- Both curl calls return `200`. The second response body includes `"dedupe":"duplicate"`.
- `updated_at` is the same value as before the second curl call (no second write).
- Only one row exists in `webhook_deliveries` with that `svix_id`.

---

## TEST 5 , Bad signature is rejected with 401

**Proves:** the Svix verification step refuses tampered payloads, preventing forged status flips.

**Run:** repeat TEST 4's curl, but flip one character in the `svix-signature` header.

**Expect:**
- HTTP `401`. No DB change.
- No row inserted into `webhook_deliveries`.

---

## TEST 6 , Unknown event type returns 200 (DEFERRED)

**Status:** Deferred manual verification. The endpoint was subscribed only to `email.sent`, `email.delivered`, `email.bounced`, `email.complained` , Resend will not send other event types to us, so the unknown-event-type path cannot be exercised end-to-end without first subscribing to an additional event we do not handle. Covered by the unit test `route.test.ts` , `returns 200 and ignores unknown event types (no DB update)`.

If you ever add a new subscribed event type to the Resend endpoint (e.g. `email.opened` for analytics), retroactively verify the behaviour by checking that any received event of the unhandled type appears in the Resend event log with response body `{"ok":true,"ignored":"<type>"}` and no `email_events` row was modified.

---

## TEST 7 , Post-delivery complaint overwrites `delivered` (DEFERRED)

**Status:** Deferred manual verification, same reason as TEST 3 , no clean way to trigger a real `email.complained` event. Covered by the unit test `route.test.ts` , `allows 'complained' to overwrite 'delivered' (post-delivery complaint)`.

Verify retroactively when the first real spam complaint arrives in production for a previously-delivered email.
