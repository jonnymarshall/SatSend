# v1.4.14 — Fiat Payment Flow + Manual Confirmation: Manual Test Guide

## What this version does

Adds an off-chain (fiat) payment path to the public invoice page and an owner-side confirm/dispute workflow. Mark-as-unpaid is now gated by *how* the invoice was confirmed — manual confirmations can be reverted, on-chain ones cannot.

Concretely:

1. **Payer flow.** Public page now shows a "Pay with USD" button alongside "Pay with Bitcoin". Clicking opens a confirmation dialog and, on confirm, transitions the invoice to a new `marked_as_paid` status with `payment_method = 'fiat'`, `payment_confirmation_method = 'manual'`. The owner gets a new email variant.
2. **Owner flow.** A `marked_as_paid` invoice shows a dedicated "Awaiting Confirmation" badge. The detail page and per-row dropdown both expose **Confirm payment received** (→ `paid`) and **Dispute / revert** (→ `pending`).
3. **Mark-as-unpaid gating.** The "Mark as unpaid" button only renders when `payment_confirmation_method = 'manual'`. On-chain-confirmed and legacy (null-method) paid invoices hide the button entirely. Reverting an on-chain payment would re-detect on the next cron sweep — the safe revert flow requires also replacing the BTC address (deferred).
4. **Confirmation-method stamping.** Every path that lands an invoice in `paid` now stamps the new fields:
   - `markPaid` / `bulkMarkPaid` (owner clicked Mark as paid) → `payment_confirmation_method = 'manual'`, `paid_at = now()`.
   - On-chain detection (cron sweep + `/api/invoices/<id>/payment-status`) → `payment_method = 'bitcoin'`, `payment_confirmation_method = 'onchain'`, `paid_at = now()`.
   - Owner confirms a `marked_as_paid` → preserves the existing `payment_method` / `payment_confirmation_method` (set when payer marked) and stamps `paid_at`.

Two migrations: `0015_fiat_and_manual_confirmation.sql` (new status, two enums, three columns) and `0016_payment_confirmed_event_type.sql` (new activity event type for owner-confirm).

## Setup

```bash
set -a; source .env; set +a
```

You will need:
- The dev server running (`npm run dev`).
- A way to query Supabase (CLI, dashboard SQL editor, or psql) — see TEST 8 in v1.4.13's doc for connection options.
- Two browser sessions: one for the **owner** (signed in to the dashboard) and one for the **payer** (private/incognito window on the public invoice page).

A few tests touch the existing on-chain path; for those you'll want either a real testnet payment or a manual SQL flip — both are noted inline.

## TEST 1 — Schema migrations are applied

Sanity check the DB state before any UI work.

**Steps:**

```sql
-- enum values
SELECT enumlabel FROM pg_enum
WHERE enumtypid = 'invoice_status'::regtype
ORDER BY enumsortorder;

SELECT enumlabel FROM pg_enum
WHERE enumtypid = 'payment_method'::regtype
ORDER BY enumsortorder;

SELECT enumlabel FROM pg_enum
WHERE enumtypid = 'payment_confirmation_method'::regtype
ORDER BY enumsortorder;

SELECT enumlabel FROM pg_enum
WHERE enumtypid = 'invoice_event_type'::regtype
ORDER BY enumsortorder;

-- columns
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_name = 'invoices'
  AND column_name IN ('payment_method', 'payment_confirmation_method', 'paid_at');
```

**Expected:**
- `invoice_status` includes `marked_as_paid`.
- `payment_method` enum: `bitcoin`, `fiat`, `bitcoin_offchain`.
- `payment_confirmation_method` enum: `onchain`, `manual`.
- `invoice_event_type` enum includes `payment_confirmed`.
- `invoices` has `payment_method` (USER-DEFINED), `payment_confirmation_method` (USER-DEFINED), `paid_at` (timestamp with time zone).

## TEST 2 — Payer marks a fiat invoice as paid (happy path)

Headline test for the new flow.

**Steps:**
1. As owner, publish a draft invoice (BTC enabled or not — both should work). Note the invoice ID.
2. In a private/incognito window, open `/invoice/<id>`. Enter the access code if required.
3. Confirm you see two payment sections: **"Pay with Bitcoin"** (if accepts_bitcoin) and the new **"Pay with USD"** card with the explanatory copy.
4. Click **Pay with USD**.
5. The dialog should display the prescribed copy verbatim:
   > "By clicking confirm, you are marking this invoice as paid. To avoid any confusion with the payee, please do not click confirm until after you have made payment."
6. Click **Confirm**.

**Expected:**
- The dialog closes and the page status updates to **Awaiting Confirmation** (purple badge). The "Pay with USD" section disappears.
- DB row: `status = 'marked_as_paid'`, `payment_method = 'fiat'`, `payment_confirmation_method = 'manual'`. `paid_at` is still `NULL` at this stage.
- An email goes out to the owner with subject `"<ClientName> marked invoice <number> as paid in USD"`. Check the Resend dashboard or the `email_events` table:
  ```sql
  SELECT email_type, recipient, status, created_at
  FROM email_events
  WHERE invoice_id = '<id>'
  ORDER BY created_at DESC;
  ```
  Most recent row should be `email_type = 'invoice_marked_paid_by_payer'`, `status = 'sent'` (or `skipped_no_api_key` in dev without `RESEND_API_KEY`).
- An activity event lands: `SELECT event_type FROM invoice_events WHERE invoice_id = '<id>' ORDER BY created_at DESC LIMIT 1;` → `marked_as_paid`.

## TEST 3 — Owner confirms a marked_as_paid invoice

**Steps:**
1. Continue from TEST 2 (or set up a fresh `marked_as_paid` invoice).
2. As owner, open the dashboard `/invoices`. The row should show the **Awaiting Confirmation** badge (purple).
3. Open the row's `⋯` menu. You should see two new items: **Confirm payment received** and **Dispute / revert** (the latter in destructive red). The regular "Mark as paid" item should be **absent** for this status.
4. Click **Confirm payment received**.

**Expected:**
- Row flips to **Paid** (green).
- DB: `status = 'paid'`, `payment_method = 'fiat'` (preserved), `payment_confirmation_method = 'manual'` (preserved), `paid_at` populated with a recent timestamp.
- Activity feed (open the detail page) shows two rows in order: the payer's "Client marked as paid" (green check) and the owner's "Payment confirmed by you" (shield).

**Same flow on the detail page:** for a `marked_as_paid` invoice, `/invoices/<id>` shows the same two buttons (`Confirm payment received` and `Dispute / revert`) instead of the usual "Mark as" menu. Confirming there should produce the same result.

## TEST 6 — Mark-as-unpaid is HIDDEN for on-chain-confirmed paid invoices

The headline gate test. Pre-v1.4.14, on-chain-confirmed invoices could be reverted, which would silently re-detect on the next cron sweep.

**Setup options — pick one:**

**a) Real on-chain payment (preferred):** publish a BTC-accepting invoice, broadcast a testnet tx, wait for confirmation. The cron sweep transitions it to `paid` with `payment_confirmation_method = 'onchain'`.

**b) Manual SQL flip (faster):** publish any invoice, then simulate the on-chain stamp:
```sql
UPDATE invoices
SET status = 'paid',
    payment_method = 'bitcoin',
    payment_confirmation_method = 'onchain',
    paid_at = now()
WHERE id = '<id>';
```

**Steps:**
1. Confirm in DB: `payment_confirmation_method = 'onchain'`.
2. As owner, open the detail page. Click the **Mark as** dropdown.
3. Open the per-row dropdown on `/invoices` for the same invoice.

**Expected:**
- Detail page **Mark as** menu shows neither **Paid** nor **Unpaid** — only **Overdue** (if eligible). No tooltip; the option is hidden entirely.
- Per-row dropdown: same — no "Mark as unpaid" item.
- For sanity, even if you bypass the UI by calling the server action directly, it must throw. Run in a Node/REPL or via a unit test:
  ```ts
  await markUnpaid("<id>");
  // → throws Error("on-chain payments cannot be reverted — replace the BTC address first")
  ```

## TEST 9 — Idempotency: payer double-clicking Confirm doesn't double-email

The server action uses a conditional update (`.eq id, .eq prev_status`); whichever submit lands second affects 0 rows and skips the email.

**Steps:**
1. Set up a fresh pending invoice in a private window.
2. Open the **Pay with USD** dialog. With network throttled (DevTools → Network → "Slow 3G"), click **Confirm** twice in rapid succession.

**Expected:**
- The dialog closes once; only one transition fires.
- `email_events` table: exactly **one** row with `email_type = 'invoice_marked_paid_by_payer'`. (Pre-fix you'd see two.)
- `invoice_events` table: exactly **one** `marked_as_paid` row.

## TEST 10 — Access-code-protected invoice still requires the cookie

The fiat-mark action re-verifies access codes server-side. A payer who somehow POSTs without going through `verifyAccessCode` should be rejected.

**Steps:**
1. As owner, publish an invoice WITH an access code set.
2. In a private window, open `/invoice/<id>` directly. The access-code gate appears — DO NOT submit it.
3. Open DevTools → Application → Cookies. Confirm there is no `pb_access_<id>` cookie.
4. Open the React DevTools console (or any way to invoke a server action manually). Try to call the action without the cookie — easiest: Edit the URL to bypass the gate, then quickly click Pay with USD before being kicked. **Or**, simulate via curl with no cookie:
   ```bash
   # No header → server action contract enforces access check.
   ```

**Expected (server-side):**
- The action returns `{ ok: false, reason: "Access code required" }`.
- The invoice status remains `pending`.
- No email goes out.

(Through the normal UI, this is hard to trigger because the access-code gate redirects you away from the page entirely. The unit test in `actions.test.ts` covers this code path — this manual test is mainly a cookie-removal sanity check.)

## Rollback notes

If you need to roll v1.4.14 back:
- Two migrations were added (0015, 0016). Reverting them requires manual SQL since enum-value drops are non-trivial:
  - You can leave the new enum values in place; nothing reads them after rollback.
  - The new columns (`payment_method`, `payment_confirmation_method`, `paid_at`) can be left in place too — old code paths don't write them.
- Reverting the merge commit fully restores prior application behaviour.
- Pre-v1.4.14 paid invoices remain unaffected (their fields stay NULL).
- Any invoices that reached `marked_as_paid` while v1.4.14 was live will need manual cleanup post-rollback (move them to `paid` or `pending` directly), since the old code doesn't recognise the status.
