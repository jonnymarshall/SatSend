# v1.4.17 — Invoices Pagination State Preserved on Navigate-Away: Manual Test Guide

## What this version does

Preserves `/invoices` pagination position across navigate-away + return. Two changes:

1. **`InvoiceDataTable` reflects pageIndex in the URL.** Page 2 onward is `?page=2`, `?page=3`, etc. Page 1 has no query param (canonical clean URL). Initial state reads from the URL on mount; `Next` / `Previous` buttons call `router.replace(...)` so the URL stays in sync without adding history entries.
2. **`← Invoices` back link uses `router.back()` with a `/invoices` fallback.** Previously a hard `<Link href="/invoices">` that always landed on page 1. Now: if there's prior history (the common case where the user came from the list), `router.back()` returns them to wherever they were including any `?page=N`. If they arrived via deep link (paste URL, email, new tab), it falls back to `/invoices`.

Out of scope (deferred): persisting global filter, sort, or archive-toggle in the URL. The bug report was about pagination specifically; those are query operations and behave like most apps already (reset on navigate).

## Setup

```bash
set -a; source .env; set +a
```

You will need:

- The dev server running (`npm run dev`).
- A logged-in dashboard tab.
- **At least 11 invoices** on your account (default page size is 10 — without enough rows there are no extra pages to test). If you don't have 11+, duplicate a few existing invoices via the row action until you do.

## Quick smoke test (~30 seconds)

If you only have time for two tests, run **TEST 1** (click-back path) and **TEST 3** (hard refresh). They cover the two halves of the fix: the back-link change and the URL-state change.

---

## TEST 1 — Click-back path: `← Invoices` returns to page 2

**Proves:** the `BackToInvoices` `router.back()` change preserves page state when the user clicks the in-app back link.

**Setup:** open `/invoices`. Confirm pagination footer shows `Previous` / `Next` buttons (i.e., there's more than one page).

**Run:**
1. Click `Next` to go to page 2. Confirm the URL now reads `/invoices?page=2`.
2. Click any invoice row's link to open `/invoices/<id>`.
3. On the detail page, click `← Invoices`.

**Expect:**
- You land back on `/invoices?page=2`, viewing the same page you came from.
- The list shows the page-2 rows, not page-1.
- No console errors.

---

## TEST 2 — Browser back button: same outcome

**Proves:** URL state alone (independent of the back-link change) survives the browser's native back navigation.

**Setup:** same as TEST 1.

**Run:**
1. Navigate to `/invoices?page=2`.
2. Click an invoice to open the detail page.
3. Press the browser's **back button** (top-left, or `Cmd+[` on macOS).

**Expect:** lands on `/invoices?page=2`, page 2 visible.

---

## TEST 3 — Hard refresh preserves the page

**Proves:** the URL is the source of truth for page state, not in-memory React state.

**Setup:** browser with the `/invoices` route already loaded.

**Run:**
1. Navigate to `/invoices?page=2`.
2. Hard refresh the tab (`Cmd+Shift+R`).

**Expect:**
- Lands back on page 2 immediately. URL still reads `/invoices?page=2`.
- No flash of page 1.
- No console errors.

---

## TEST 4 — Direct URL share

**Proves:** shareability — pasting a URL with `?page=N` lands on that page in a fresh tab.

**Setup:** any browser session signed in to your account.

**Run:**
1. Open a fresh tab.
2. Paste `<your-app-url>/invoices?page=2` into the address bar and press enter.

**Expect:** you land directly on page 2 without first flashing page 1.

---

## TEST 5 — Out-of-bounds `?page=` clamps gracefully

**Proves:** pasting `?page=99` (or any value beyond the last page) does not crash or render an empty table — it clamps to the last available page.

**Setup:** any browser session signed in.

**Run:**
1. Open `/invoices?page=99` (assuming you have far fewer than 99 pages).

**Expect:**
- The table renders the **last available page** (page 2 if you have 11–20 invoices).
- The URL updates to reflect the clamped value (e.g. `/invoices?page=2`) — TanStack's `setPageIndex(getPageCount() - 1)` flows through `onPaginationChange` which writes the URL.
- No empty-table flash, no console errors.

---

## TEST 6 — Garbage `?page=` value defaults to page 1

**Proves:** defensive parsing handles malformed query strings without crashing.

**Run:**
For each of the following URLs, open in a fresh tab and confirm the result:

| URL | Expected page |
|-----|---------------|
| `/invoices?page=foo` | page 1 |
| `/invoices?page=0` | page 1 |
| `/invoices?page=-1` | page 1 |
| `/invoices?page=` (empty) | page 1 |

**Expect:** all four land on page 1 without console errors. The URL keeps the malformed param (we don't try to "clean up" garbage URLs proactively); subsequent pagination clicks will rewrite it correctly.

---

## TEST 7 — Sidebar nav doesn't carry stale `?page=`

**Proves:** we haven't accidentally over-persisted state — clicking "Invoices" in the sidebar from another route lands on a clean page-1 view, not whatever page you were last on.

**Run:**
1. Navigate to `/invoices?page=2`.
2. Click into another route (e.g. `/settings` or whatever your sidebar offers).
3. Click the sidebar's **Invoices** link.

**Expect:**
- Lands on bare `/invoices` (no `?page=`).
- View shows page 1.
- This is the intended behavior: the sidebar nav is a "fresh visit", not a "go back".

---

## TEST 8 — Deep-link back-link falls back to `/invoices`

**Proves:** the `router.back()` fallback path works when there's no prior history (e.g. the user pasted an invoice URL into a new tab).

**Run:**
1. Copy an invoice detail URL (e.g. `/invoices/<id>`).
2. Open a brand-new tab (no other history in this tab).
3. Paste the URL and load it.
4. On the detail page, click `← Invoices`.

**Expect:**
- Lands on `/invoices` (page 1, no query string).
- Does not stay on the detail page (which would happen if `router.back()` had nowhere to go and silently no-op'd — the fallback prevents that).
- No console errors.
